# mm_init 완벽 정리 (심화 버전)

---

## 1️⃣ 전체 구조 한눈에

```
mm_init이 만드는 초기 힙 구조:

┌────────┬──────────────────┬──────────┬──────────────────┬────────┐
│ 패딩   │ 프롤로그 블록    │          │  가용 블록 (4KB)  │에필로그│
│        ├──────────────────┤          ├──────────────────┤        │
│ (4B)   │ 헤더(9) │푸터(9)│          │ 헤더(4096/0)│푸터│(1)    │
└────────┴──────────────────┴──────────┴──────────────────┴────────┘
0x100   0x104   0x108    0x10C        0x110          ...      0x1000
         ↑                                                      ↑
    heap_listp                                          최초 힙 끝
```

---

## 2️⃣ 각 부분 역할 정리

### 패딩 (Padding)

**역할**: 메모리 정렬

```c
PUT(heap_listp, 0);  // 그냥 0, 의미 없음
```

**왜 필요한가?**
- 8바이트 정렬 조건 충족
- 페이로드가 8의 배수 주소에서 시작하도록
- CPU 캐시 효율 향상 → 속도 빨라짐 ⚡

**숨겨진 이유**
- `heap_listp`가 프롤로그 블록의 페이로드 위치를 가리키기 위함
- 프롤로그도 최소 블록 크기(16B) 조건을 만족하기 위함

---

### 프롤로그 (Prologue)

**역할**: 힙 시작 경계 보호

```c
PUT(heap_listp + (1 * WSIZE), PACK(DSIZE, 1));  // 헤더: 9 (크기8, 할당1)
PUT(heap_listp + (2 * WSIZE), PACK(DSIZE, 1));  // 푸터: 9 (크기8, 할당1)
```

**구조**
```
프롤로그는 특이한 블록:
- 페이로드 없음 (크기 = 헤더 + 푸터 = 8B)
- 헤더 = 푸터 (모두 9)
- 항상 "할당됨(1)" 상태 유지
```

**왜 필요한가?**

coalesce에서 첫 번째 블록을 free할 때:

```c
size_t prev_alloc = GET_ALLOC(FTRP(PREV_BLKP(bp)));
```

이 코드가 **첫 번째 블록의 이전 블록**을 찾으려고 함.

프롤로그 없으면:
```
[첫번째 블록]
     ↑
PREV_BLKP(첫번째) → 메모리 밖 💥
                     존재하지 않는 주소!
                     프로그램 터짐!
```

프롤로그 있으면:
```
[프롤로그][첫번째 블록]
    ↑
PREV_BLKP(첫번째) → 프롤로그 ✅
                     안전한 위치!
                     할당됨(1) → 합체 안 함
```

**핵심**: coalesce가 힙 바깥으로 튀어나가지 않게 막는 경계 파수꾼!

---

### 에필로그 (Epilogue)

**역할**: 힙 끝 표시

```c
PUT(heap_listp + (3 * WSIZE), PACK(0, 1));  // 헤더: 1 (크기0, 할당1)
```

**특징**
- 크기 = 0 (더 이상 블록이 없다는 신호)
- 헤더만 있음 (푸터 없음)
- 항상 "할당됨(1)" 상태

**왜 필요한가?**

find_fit의 for문:

```c
for (bp = heap_listp; GET_SIZE(HDRP(bp)) > 0; bp = NEXT_BLKP(bp)) {
    if (!GET_ALLOC(HDRP(bp)) && asize <= GET_SIZE(HDRP(bp))) {
        return bp;
    }
}
```

에필로그 없으면:
```
[프롤로그][블록1][블록2][...]???
                        ↑
                   어디서 끝나?
                   무한루프 💥
```

에필로그 있으면:
```
[프롤로그][블록1][블록2][...][에필로그(크기=0)]
                                  ↑
                             GET_SIZE = 0
                             → for문 조건 FALSE
                             → 루프 끝! ✅
```

**핵심**: find_fit 루프가 안전하게 끝나도록 탈출 신호 제공!

## 3️⃣ coalesce 5가지 상황

### 상황 분석 과정

```c
size_t prev_alloc = GET_ALLOC(FTRP(PREV_BLKP(bp)));  // 이전 블록 상태
size_t next_alloc = GET_ALLOC(HDRP(NEXT_BLKP(bp)));  // 다음 블록 상태
```

이 두 값으로 4가지 경우가 생김:
- (1, 1): 양쪽 할당
- (1, 0): 이전 할당, 다음 가용
- (0, 1): 이전 가용, 다음 할당
- (0, 0): 양쪽 가용

---

### Case 1: prev_alloc=1, next_alloc=1 (양쪽 할당)

```
Before: [할당][현재(free됨)][할당]

Action: 아무것도 안 함

After: [할당][가용][할당]
            ↑
        그냥 return bp
```

**왜?** 합칠 게 없으니까 비용 낭비 금지!

---

### Case 2: prev_alloc=1, next_alloc=0 (다음만 가용)

```
Before: [할당][현재 16B][다음 32B]
               헤더→16/0  헤더→32/0
               ...        ...
               푸터→16/0  푸터→32/0

Action:
size = 16 + 32 = 48
PUT(HDRP(bp), PACK(48, 0))      // 현재 헤더 갱신
PUT(FTRP(bp), PACK(48, 0))      // 다음 블록의 푸터 위치에 쓰기!

After: [할당][      가용 48B      ]
             헤더→48/0
             ...
             푸터→48/0 (다음 블록 끝)

포인터: bp (변경 안 함)
```

**핵심**: 다음 블록의 푸터가 자동으로 현재 블록의 푸터가 됨
(FTRP(bp) 매크로가 GET_SIZE(HDRP(bp))를 사용하니까)

---

### Case 3: prev_alloc=0, next_alloc=1 (이전만 가용)

```
Before: [이전 32B][현재 16B][할당]
        헤더→32/0 헤더→16/0
        ...       ...
        푸터→32/0 푸터→16/0

Action:
size = 32 + 16 = 48
PUT(FTRP(bp), PACK(48, 0))           // 현재 푸터 갱신
PUT(HDRP(PREV_BLKP(bp)), PACK(48, 0)) // 이전 헤더 갱신
bp = PREV_BLKP(bp)                     // bp 이동!

After: [      가용 48B      ][할당]
       헤더→48/0
       ...
       푸터→48/0

포인터: bp가 이전 블록으로 이동! (중요!)
```

**핵심**: bp를 이전 블록 시작점으로 이동
(합쳐진 블록의 시작이 이전 블록이니까)

---

### Case 4: prev_alloc=0, next_alloc=0 (양쪽 가용)

```
Before: [이전 16B][현재 16B][다음 16B]
        헤더→16/0 헤더→16/0 헤더→16/0
        ...       ...       ...
        푸터→16/0 푸터→16/0 푸터→16/0

Action:
size = 16 + 16 + 16 = 48
PUT(HDRP(PREV_BLKP(bp)), PACK(48, 0))  // 이전 헤더 갱신
PUT(FTRP(NEXT_BLKP(bp)), PACK(48, 0))  // 다음 푸터 갱신
bp = PREV_BLKP(bp)                      // bp 이동!

After: [         가용 48B         ]
       헤더→48/0
       ...
       푸터→48/0

포인터: bp가 이전 블록으로 이동!
```

**핵심**: 세 블록을 하나로 합침 (최대 효율!)

---

## 4️⃣ 왜 헤더와 푸터를 쌍으로 쓰나?

### 헤더만 있으면 안 되냐?

**문제 1**: 이전 블록의 크기를 모름

```c
PREV_BLKP(bp) = bp - GET_SIZE((char*)(bp) - DSIZE)
                 ↑ 이전 블록의 푸터를 읽어야 함!
```

**문제 2**: coalesce할 때 후반부 블록을 찾을 수 없음

```c
FTRP(bp) = bp + GET_SIZE(HDRP(bp)) - DSIZE
           ↑ 이전 블록 푸터가 있어야만 이전 블록 크기를 알 수 있음
```

**해결책**: 헤더 + 푸터를 양쪽 끝에 배치
- 앞에서는 헤더로 크기 파악
- 뒤에서는 푸터로 크기 파악
- 양쪽 방향으로 자유롭게 네비게이션 가능!

---

## 5️⃣ 메모리 단편화 문제

### 문제: 외부 단편화 (External Fragmentation)

coalesce 없이 계속 malloc/free 반복:

```
[가용8B][할당][가용8B][할당][가용8B][할당][가용8B]
         ↑
       32B 요청 들어옴

각각 8B짜리인데... 연속된 16B도 없음! 💥
실제로는 32B가 비어있는데도 할당 실패!
```

### 해결책: coalesce로 블록 합치기

```
free할 때마다 옆 블록과 합치기:

[      가용 16B      ][할당][      가용 16B      ][할당]

→ 다음 malloc이 온다면 16B 블록 찾기 쉬움!
→ 필요하면 합친 블록 여러 개 사용 가능
```

## 6️⃣ 프롤로그/에필로그의 본질

### 디자인 패턴: Sentinel Pattern (파수꾼 패턴)

```
일반적인 처리 (복잡):
if (bp == heap_start) {
    // 첫 번째 블록, 특별 처리
    prev = NULL;
} else {
    prev = PREV_BLKP(bp);
}

파수꾼 패턴 (단순):
if (prev_alloc) {
    // 항상 같은 코드!
    // 프롤로그가 첫 블록의 "이전"으로 기능
}
```

**장점**:
- Edge case 없음 (첫 블록/마지막 블록 예외 처리 불필요)
- 코드 단순화
- 버그 가능성 감소

---

## 7️⃣ 정렬의 중요성

### 왜 8바이트 정렬?

```
64비트 CPU 기준:
- 워드 크기 = 8바이트
- 캐시 라인 = 보통 64바이트 (8의 배수)
- 메모리 버스 = 8바이트 단위

8바이트 정렬:
- 한 번의 메모리 접근으로 완전한 데이터 읽음
- CPU 사이클 감소
- 성능 향상 ⚡
```

**정렬 안 되면:**
```
0x109에서 8바이트 읽으면?
CPU가 0x108~0x110과 0x110~0x118 두 번 읽음 (Misaligned Access)
→ 버스 접근 증가
→ 성능 저하
```

## 8️⃣ 전체 흐름 요약

```
1. mem_sbrk(16) → 16바이트 확보
   ↓
2. PUT으로 4개 워드 초기화
   - 패딩(0)
   - 프롤로그 헤더(9)
   - 프롤로그 푸터(9)
   - 에필로그 헤더(1)
   ↓
3. heap_listp = 프롤로그 푸터 주소
   (모든 블록 탐색의 시작점)
   ↓
4. extend_heap → 실제 가용 블록 생성
   ↓
5. 준비 완료! malloc/free 가능 ✅
```

## 9️⃣ 핵심 개념 정리

| 개념 | 역할 | 이유 |
|------|------|------|
| **패딩** | 8바이트 정렬 | CPU 성능 최적화 |
| **프롤로그** | 힙 시작 경계 | coalesce 안전성 |
| **에필로그** | 힙 끝 표시 | find_fit 루프 종료 |
| **헤더+푸터** | 양방향 네비게이션 | 이전/다음 블록 찾기 |
| **coalesce** | 블록 합치기 | 외부 단편화 방지 |

## 🔟 실제 malloc/free 예시

### 초기 상태
```
[프롤로그][      가용 4KB      ][에필로그]
```

### malloc(100) 호출
```
[프롤로그][할당 120B][     가용 3976B      ][에필로그]
         ↑
      place() 블록 분할
```

### malloc(200) 호출
```
[프롤로그][할당 120B][할당 224B][  가용 3752B  ][에필로그]
```

### free(첫번째) 호출
```
[프롤로그][가용 120B][할당 224B][  가용 3752B  ][에필로그]
         ↑
      coalesce() → 옆 다음이 할당이니까 합체 안 함
```

### free(마지막) 호출
```
[프롤로그][가용 120B][할당 224B][       가용 3752B       ][에필로그]
                                 ↑
                                coalesce → 이전 할당이니까 합체 안 함
```

### malloc(200) 호출 (2번째)
```
[프롤로그][가용 120B][할당 224B][할당 224B][  가용 3528B  ][에필로그]
                                 ↑
                              find_fit으로 가용 3752B 찾음
                              place()로 224B 할당, 나머지 분할
```

## 마지막 정리

이 모든 설계의 목표는 **단순하면서도 안전한 메모리 관리**!

```
패딩 → 정렬 → 성능 ⚡
프롤로그 → 경계 → 안전성 🛡️
에필로그 → 종료 → 안정성 ✅
coalesce → 합체 → 효율성 🧩
```

이 네 가지가 함께 작동해서 완벽한 동적 메모리 할당기가 완성돼! 🎯
